## 🎯 项目定位与架构 (Project Identity)

`gemini-cli-telegram` 是本地 **Google Antigravity CLI (`agy`)** 的 Telegram 适配网关。它作为桥梁运行在后台守护进程中，将 Telegram Bot API 与 Google Antigravity/Gemini 底层核心引擎无缝连接。

### 1. 核心功能特征
- **项目与会话管理**：支持多项目自适应切换、原子会话重置（Atomic Session）、以及完整的历史清除。
- **高阶工具执行**：使 AI 编码助手能够在 Telegram 聊天界面中直接修改文件、执行系统命令、查询网络。
- **流式打字机**：支持流式文本与富文本输出，提供丝滑、防重发的打字状态体验。
- **多模态解析**：完美支持文本、照片、语音（自动转译）、视频、文档等多模态输入。

### 2. 本地 systemd 服务集成（独立托管）
```mermaid
graph TD
    User["Telegram 用户"] <-->|Webhook/Polling| Bot["Telegram Bot (Node.js 22)"]
    Bot <-->|onChunk Stream| MsgLoop["messageLoop (流式锁/防重发)"]

```

## 🔒 认证鉴权与安全约束 (Authentication & Security)

### 1. 本地会话借用机制 (Local Session Borrowing)
- 守护进程会优先读取和借用本地已有的交互式 `gemini-cli` 登录态，检查路径包括 `~/.gemini/gemini-credentials.json` 或项目根目录下的 `oauth_creds.json`。
- **API Key 备用**：环境变量 `GEMINI_API_KEY` 虽然支持，但若检测到上述本地登录态则会被自动忽略，以防止多项目或多账户会话冲突。

### 2. Headless 环境兼容
- 由于 Bot 运行在无图形界面的 Headless 后台守护进程中，无法触发交互式 Google 浏览器 OAuth 流程。
- **开发约束**：若需初次授权或登录态过期，用户**必须**先在交互式终端中执行如下命令生成凭据：
  ```bash
  gemini-cli-telegram setup auth
  ```

### 3. 白名单访问控制
- 机器人具备严格的安全限制，仅允许在配置文件的白名单（`allowedUsers`）中的 Telegram 账户进行私聊与指令调用，避免服务器资源被滥用。

## 🎨 消息渲染引擎与 API 10.1 升级 (Formatting & Rendering)

系统支持三种渲染解析模式，可通过项目的 `.gemini/settings.json` 进行自适应配置。其中，**RichText (API 10.1) 模式**已在最新升级中进行了重构：

### 1. 解析模式对比

| 模式 | 运行机制 | 优缺点分析 |
| :--- | :--- | :--- |
| **HTML (Default)** | 将 Markdown 格式转为 standard HTML 标签发送。 | 极度稳健，流式渲染支持最好，但缺乏复杂的嵌套布局和 Zebra 表格支持。 |
| **MarkdownV2** | 使用 Telegram 经典的 MarkdownV2 转义引擎（包含 19 个必须转义的字符）。 | 格式原生度高，但由于对闭合标签和转义极其敏感，容易因转义不全导致整条消息报错。 |
| **RichText (API 10.1)** | 采用 API 10.1 新增富文本架构，通过 `sendRichMessage` 传输原生高级图表。 | **黄金推荐**：支持原生斑马纹表格、 details 块、 LaTeX 公式。最新已彻底修复旧版 Option A 发送 blocks 时 100% 报错的 Bug。 |

### 2. API 10.1 三级回退安全链路 (Highly Resilient Pipeline)
为了保证 100% 消息发送成功，在 `parseMode === 'RichText'` 下采用以下渐进降级策略：
- **Option A (首选 HTML 模式)**：调用 `sendRichMessage` 传入 `{ html: markdownToHtml(originalText) }`。因为 Telegram 服务端会在云端将其解析为完全一致的富文本块，这不仅完全修复了旧版直接发送 blocks AST 时的 API 崩溃 bug，更免去了本地脆弱的 AST 解析开销。
- **Option B (备选 Markdown 模式)**：若 HTML 解析抛出边缘异常，立即自动降级并发送 `{ markdown: prepareTelegramMarkdown(originalText) }` 格式。
- **Option C (最坏情况兜底)**：若 API 10.1 链路由于外界或 Telegram 服务器突发异常故障，则通过传统的 `parse_mode: 'HTML'` 方式使用 `ctx.reply` 发送。

### 3. 打字机流式草稿与原生动画 `<tg-thinking>`
- **草稿路由重定向**：在富文本流式渲染下，底层的纯文本输出 `sendPlain` / `editPlain` 均已被自动路由至临时草稿写信接口 `sendRichDraft`（调用 Telegram 的 `sendRichMessageDraft`）。
- **原生思考动画**：当进行流式输出且处于思考/生成阶段时，系统会自动在 HTML 块末尾附带 **`<tg-thinking>Thinking...</tg-thinking>`** 标签。这会在 Telegram 手机端/PC端展示极具质感的官方原生思考打字动效，大幅提升交互体验。

## 🚀 关键指令与交互逻辑 (Key Commands & Logic)

- **`/model <name>`**：在 11 种预置模型（Gemini 3.5/3.1/2.5 和 Gemma 4）以及 `Web2API:` 逆向网页端模型（需配置 Python 服务端）之间一键切换。
- **`/save`（知识采集归档）**：对機器人回复的格式化消息进行快捷采集，保存至知识库（默认为 `/mnt/pool/1000/jack/00_输入缓冲_Inbox/外部采集_External`）。
  - **命名规范**：抽取回复原文的第一行 `## 标题` 作为文件名。
  - **文件名约束**：`YYYYMMDD_安全Title.md`（中文标题限制在 15 个字内，多余会被安全裁剪并剔除特殊标点）。
- **`/undo`**：从当前会话的活跃上下文中撤销上一次用户和助理的对话交互。
- **`/delete_session <index>`**：从服务器磁盘上彻底物理删除指定的旧会话文件。
- **`/new`**：即刻重置当前聊天，开启全新的原子会话上下文。
- **消息临时缓存**：自动在本地维持一个 `1000条消息 / 24小时` 的内存滑动窗口缓存，保存未格式化的原始 Markdown 源码，以供 `/save` 指令精确抽取。
- **定时会话清理**：在服务器中设置系统 Cron，每日定时触发 `scripts/cleanup_sessions.sh`，物理擦除超过 30 天无互动的闲置 session 文件。

## 💻 运维管理指南 (Maintenance & Run)

所有源码均位于 `src/` 中。请注意：**严禁直接修改 `dist/` 中的编译产物**，必须通过 TypeScript 编译更新。

### 1. 常用管理指令
在部署目录 `/home/qcsunny/.gemini-cli-telegram` 下：

```bash
# 1. 静态类型检查与项目编译（生成最新 dist）
npm run build

# 2. 启动 Telegram 守护进程（后台运行）
node dist/cli.js start

# 3. 停止当前运行的守护进程
node dist/cli.js stop

# 4. 检查守护进程运行状态
node dist/cli.js status

# 5. 实时监控守护进程日志
node dist/cli.js logs
# 或直接 tail 观察
tail -f daemon.log
```

### 2. 全局代理链
所有的外部请求（包括 Telegram API 交互、媒体资源下载、Google Auth 投征、Gemini API 请求等）均通过本地配置文件 `config.json` 中配置的本地代理（通常为本地 Clash `http://127.0.0.1:7890`）进行中转：
- **Bot API**：依靠 grammY 底层接入的 `HttpsProxyAgent` 保持稳定长连接。
- **全局网络**：在配置加载之初即调用 `undici.setGlobalDispatcher` 将代理应用至全局 fetch 中，防止早期的认证探针请求被墙阻塞。

## 🛠️ 项目开发与 Commit 规范 (CLAUDE.md Rules)

为了保持项目交付质量，团队采用高度规范的测试与提交流程。**所有提交（Commit）必须包含测试方案说明。**

### 1. Commit 格式要求
每次提交的 Commit Message 必须在正文（Body）部分明确包含一个 **"How to test"（如何验证）** 章节：
- 提供可打开和验证更改的 Live 测试链接（如有）；
- 逐步写明需要点击/校验的交互流程；
- 提供登录所需的测试凭据（若有鉴权拦截）；
- 写明每一个操作步骤的预期结果（Expected Result）。

### 2. 标准提示例
```
feat: 增加对 Telegram 10.1 富文本 API 的 HTML 传输支持

How to test:
- 启动本地 Telegram 守护进程：node dist/cli.js start
- 在测试群组中向 Bot 发送一条包含复杂 GFM 表格和 details 折叠块的消息：
  "## 性能对比
   | 模型 | 耗时 |
   |---|---|
   | Flash | 100ms |"
- 预期：在手机端能够正确渲染出带圆角和斑马纹的 Telegram 原生表格。
- 观察本地 daemon.log：确认无 fallback Option B/C 的警告信息。
```

## 🧼 Superpowers-ZH 中文增强框架

本项目集成了 superpowers-zh 生产力增强框架，共装备 20 个可用 skills。

### 1. 开发四大核心规则
- **检查 Skill 匹配**：收到任何开发任务时，1% 的可能性也要先遍历本地技能目录 `.gemini/skills/` 检查是否有现成的方法论参考。
- **设计先于编码**：在着手重写代码或新建组件前，先调用 `brainstorming` 技能，明确用户诉求与设计边界。
- **测试先于实现 (TDD)**：编写任何实现逻辑前，优先补充测试用例，运行测试发现红灯，写完代码使测试全绿。
- **验证先于完成**：声称重构完毕或 Bug 修复前，必须运行实际的集成测试与验证指令，用绿灯作为成功的客观证据。

### 2. 部分核心 Skills 目录与定位

- **`brainstorming`**：在创造性实现（创建功能、组件、逻辑）之前运行，深度对齐意图。
- **`systematic-debugging`**：遇到报错、测试失败或死锁时的系统化排查和日志分析指南。
- **`test-driven-development`**：在编写实际逻辑代码前规范 TDD 流程。
- **`verification-before-completion`**：交付、提交 commit 或发起 PR 前的验证和证据核对指南。
- **`writing-plans`**：当任务横跨多文件、多层次时的实现步骤详细拆解。
